## Prerequisites

- A **Railway account** (a paid plan is recommended — the stack runs several always-on services).
- An **SMTP provider** (or Zoho/ZeptoMail) for signup verification, password reset, and invite emails. See [SMTP Setup](#smtp-setup).
- *Optional:* a **custom domain** for the dashboard and storage, an **OpenAI API key** (transcription + AI course generation), and Google OAuth / Unsplash keys.

<Callout title="Two secrets to generate first">
  You'll reuse `BETTER_AUTH_SECRET` and `PRIVATE_SERVER_KEY` across services. Generate them once and keep them handy:

  ```zsh title="Terminal"
  openssl rand -hex 32   # BETTER_AUTH_SECRET
  openssl rand -hex 32   # PRIVATE_SERVER_KEY  (must be identical in cio-api and cio-dashboard)
  ```
</Callout>

## Step-by-step walkthrough

The one-click template provisions **all of the services** for you — Postgres, Redis, `cio-api`, `cio-dashboard`, `cio-jobs`, and `cio-minio` (object storage). The flow is: **configure a few values → deploy → connect the dashboard domain → sign up**.


<Steps>

<Step>
### Click "Deploy on Railway"

Click the **Deploy on Railway** button above. Railway opens the template page where all of the ClassroomIO services are listed and ready to configure.

![Railway template page for ClassroomIO after clicking Deploy](/railway-01-deploy-template.png)
</Step>

<Step>
### Review the preconfigured services

The template comes with every service preconfigured — most need nothing from you:

- **Redis** and **Postgres** — no configuration required.
- **cio-dashboard** — no required configuration (optionally set branding such as the public OG image; not required).

The next three steps cover the services that **do** need input: `cio-jobs`, `cio-minio`, and `cio-api`.

![The Railway template listing all preconfigured ClassroomIO services](/railway-02-services-overview.png)
</Step>

<Step>
### Configure cio-jobs (SMTP)

`cio-jobs` sends invites, notifications, verification, and password-reset emails, so it needs SMTP credentials. Fill these in:

- `SMTP_HOST`
- `SMTP_PORT`
- `SMTP_USER`
- `SMTP_SENDER`
- `SMTP_PASSWORD`

![Entering the SMTP variables on the cio-jobs service in the template](/railway-03-jobs-smtp.png)
</Step>

<Step>
### Configure cio-minio (object storage)

Set the storage credentials on `cio-minio`:

- `MINIO_SERVER_URL`
- `MINIO_ROOT_USER`
- `MINIO_ROOT_PASSWORD`

![Configuring MINIO_SERVER_URL and credentials on the cio-minio service](/railway-04-minio-credentials.png)
</Step>

<Step>
### Configure cio-api

On `cio-api`, set the SMTP host and any additional values you want. See the [Environment variable reference](#environment-variable-reference) below for the full list (object storage, auth secrets, dashboard origin, and more).

![Configuring the cio-api service variables in the template](/railway-05-api-config.png)
![Additional cio-api configuration options](/railway-06-api-config-additional.png)
</Step>

<Step>
### Deploy

Click **Deploy**. Railway begins building and bringing the services online — Redis, Postgres, and the rest come up first.

![Railway deploying the ClassroomIO services after clicking Deploy](/railway-07-deploying.png)
</Step>

<Step>
### First boot may crash — this is expected

<Callout type="warn" title="Only if you use the Railway-generated subdomain">
  On the first deploy, `cio-api` can crash. This is expected **only when you rely on the Railway-generated subdomain**: the API needs the dashboard's public domain (Better Auth's base URL), which doesn't exist yet. If you're using your **own custom subdomain**, this crash doesn't happen — skip ahead to signing up.
</Callout>

![cio-api crashing on first boot before the dashboard domain exists](/railway-08-first-boot-crash.png)
</Step>

<Step>
### Generate the dashboard domain

Open **cio-dashboard → Settings → Networking** and click **Generate Domain**. Railway assigns the public domain the API needs.

![cio-dashboard Settings and Networking with the Generate Domain button](/railway-09-generate-domain.png)
</Step>

<Step>
### Redeploy cio-dashboard and cio-api

Redeploy **cio-dashboard**, then **cio-api**, so both pick up the newly generated domain. They should now boot cleanly.

![Redeploying the cio-dashboard](/railway-10-redeploy-dashboard.png)
![Redeploying the cio-api](/railway-11-redeploy-api.png)
</Step>

<Step>
### Open the app and sign up

Open the generated dashboard URL in a **new browser tab** and **sign up** to create your first account.

![The running ClassroomIO dashboard sign-up page on the Railway domain](/railway-12-signup.png)
</Step>

## Environment variable reference

You set these values **while configuring each service on the template page** (the configure steps above) — you don't enter them again after deploying. Use this section as the reference for what each service's variables should be.

### cio-api

```zsh title="env"
DATABASE_URL="${{db.DATABASE_URL}}"
REDIS_URL="${{Redis.REDIS_URL}}"
BETTER_AUTH_SECRET="<generate-with-openssl-rand-hex-32>"
PRIVATE_SERVER_KEY="<generate-with-openssl-rand-hex-32>"   # must be identical in cio-api and cio-dashboard
PUBLIC_IS_SELFHOSTED="true"

# Dashboard URL for invite/email links. Reference the dashboard service's Railway domain
# so it stays correct automatically. (For a custom domain, hardcode it: https://app.yourdomain.com)
DASHBOARD_ORIGIN="https://${{cio-dashboard.RAILWAY_PUBLIC_DOMAIN}}"

# Optional direct public API URL. Normal dashboard auth/API calls use the
# dashboard proxy and do not require API/dashboard cookie-domain matching.
PUBLIC_SERVER_URL="https://api.yourdomain.com"
TRUSTED_ORIGINS=

# Email (configure SMTP)
SMTP_HOST=
SMTP_PORT=
SMTP_USER=
SMTP_PASSWORD=
SMTP_SENDER=

# Object storage → the self-hosted cio-minio service (see below)
OBJECT_STORAGE_ENDPOINT=http://${{cio-minio.RAILWAY_PRIVATE_DOMAIN}}:9000
OBJECT_STORAGE_PUBLIC_ENDPOINT=https://storage.yourdomain.com
OBJECT_STORAGE_ACCESS_KEY_ID=${{cio-minio.MINIO_ROOT_USER}}
OBJECT_STORAGE_SECRET_ACCESS_KEY=${{cio-minio.MINIO_ROOT_PASSWORD}}
OBJECT_STORAGE_FORCE_PATH_STYLE=true
OBJECT_STORAGE_MEDIA_PUBLIC_BASE_URL=https://storage.yourdomain.com/media
```

Optional: add `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET` for OAuth, `UNSPLASH_API_KEY` for image search, or `LICENSE_KEY` for enterprise features.

### cio-dashboard

```zsh title="env"
# RAILWAY_PUBLIC_DOMAIN here = this dashboard service's own public domain.
# For a custom domain, hardcode app.yourdomain.com in ORIGIN / PRIVATE_APP_HOST instead.
ORIGIN="https://${{RAILWAY_PUBLIC_DOMAIN}}"
PRIVATE_APP_HOST="${{RAILWAY_PUBLIC_DOMAIN}}"
PUBLIC_IS_SELFHOSTED="true"
PRIVATE_SERVER_URL="http://${{cio-api.RAILWAY_PRIVATE_DOMAIN}}:${{cio-api.PORT}}"
PRIVATE_SERVER_KEY="${{cio-api.PRIVATE_SERVER_KEY}}"   # same value as cio-api
ALLOWED_EXTERNAL_DOMAINS="https://fonts.googleapis.com"
```

Optional: add `PUBLIC_APP_TITLE`, `PUBLIC_APP_DESCRIPTION`, `PUBLIC_OG_IMAGE_URL` for custom branding. You can also use per-directive CSP vars (`CSP_SCRIPT_SRC_DOMAINS`, `CSP_MEDIA_SRC_DOMAINS`, etc.) instead of `ALLOWED_EXTERNAL_DOMAINS` for finer control.

**Upload size limits** (optional): add the same `UPLOAD_MAX_*_MB` values to **cio-api** and **cio-dashboard** (e.g. `UPLOAD_MAX_DOCUMENT_MB=50`). See [Docker self-hosting → Upload file size limits](/docs/self-hosted/docker#optional--upload-file-size-limits).

### cio-jobs (background worker)

The jobs worker has **no public domain** — it drains the Redis queue (video processing, transcription, AI course generation, and emails). It shares the same Postgres, Redis, storage, email, and OpenAI settings as the API:

```zsh title="env"
DATABASE_URL="${{db.DATABASE_URL}}"
REDIS_URL="${{Redis.REDIS_URL}}"

# Object storage — same cio-minio service as the API
OBJECT_STORAGE_ENDPOINT=http://${{cio-minio.RAILWAY_PRIVATE_DOMAIN}}:9000
OBJECT_STORAGE_PUBLIC_ENDPOINT=https://storage.yourdomain.com
OBJECT_STORAGE_ACCESS_KEY_ID=${{cio-minio.MINIO_ROOT_USER}}
OBJECT_STORAGE_SECRET_ACCESS_KEY=${{cio-minio.MINIO_ROOT_PASSWORD}}
OBJECT_STORAGE_FORCE_PATH_STYLE=true
OBJECT_STORAGE_MEDIA_PUBLIC_BASE_URL=https://storage.yourdomain.com/media

# Email (worker sends invites, notifications, etc.)
SMTP_HOST=  SMTP_PORT=  SMTP_USER=  SMTP_PASSWORD=  SMTP_SENDER=

# Transcription + OpenAI-backed AI course generation
OPENAI_API_KEY=
```

<Callout title="PRIVATE_SERVER_KEY">
  `PRIVATE_SERVER_KEY` must be the same value in both API and dashboard. It authenticates server-side dashboard requests to the API. Referencing `${{cio-api.PRIVATE_SERVER_KEY}}` in the dashboard keeps them in sync.
</Callout>

### cio-minio (object storage)

`cio-minio` ships **in the template** — its `/data` volume is already attached and the `videos`,
`documents`, and `media` buckets are already seeded. You only set its credentials during
configuration:

```zsh title="env"
MINIO_ROOT_USER=<strong user>
MINIO_ROOT_PASSWORD=<strong password>
# Optional (silences console redirect warnings behind the proxy):
MINIO_SERVER_URL=https://storage.yourdomain.com
```

The api and jobs services reach MinIO over **two** endpoints: the private `OBJECT_STORAGE_ENDPOINT`
(server uploads/reads) and the public `OBJECT_STORAGE_PUBLIC_ENDPOINT` + `OBJECT_STORAGE_MEDIA_PUBLIC_BASE_URL`
(browser-facing presigned downloads and public images) — already wired in the env blocks above.
`OBJECT_STORAGE_FORCE_PATH_STYLE=true` is required for MinIO.

<Callout type="warn" title="MinIO is your durability boundary">
  Single-node MinIO has no redundancy — the Railway Volume is the only thing protecting your
  uploads. Back it up. If you'd rather not operate storage, use external **Cloudflare R2 / AWS S3**
  instead (R2 has no egress fees): point the same `OBJECT_STORAGE_*` vars at the provider. See the
  [Cloudflare Setup](/docs/contributor-guides/cloudflare-setup) guide for R2.
</Callout>

### SMTP Setup

You need SMTP for email sending. See [Nodemailer SMTP docs](https://nodemailer.com/smtp/) for provider setup.

### SSO (Enterprise)

SSO requires a license key. Set `LICENSE_KEY` in your API environment. [Book a call](http://cal.com/classroomio/enterprise) to get a license.
