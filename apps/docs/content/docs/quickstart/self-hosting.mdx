This guide covers three deployment paths: **Docker** (recommended), **Railway**, and **Coolify**.

![Selfhost ClassroomIO](/selfhost-classroomio.webp)

## Docker (Recommended)

### Prerequisites

- Docker Desktop installed and running
- `git` installed

### Quick Start

1. Clone the repository:

```zsh title="Terminal"
git clone https://github.com/classroomio/classroomio.git
cd classroomio
```

2. Copy the env template and edit it:

```zsh title="Terminal"
cp .env.example .env
```

Open `.env` and set `DASHBOARD_ORIGIN` and `BETTER_AUTH_SECRET` for your domain. For local development the defaults work as-is.

3. Start the full stack:

```zsh title="Terminal"
./run-docker-full-stack.sh
```

The script builds and starts Postgres, Redis, MinIO (object storage), the API, and the dashboard. It auto-generates secure values for `AUTH_BEARER_TOKEN` and `PRIVATE_SERVER_KEY` when they are missing.

4. Verify:

```zsh title="Terminal"
curl -sS http://localhost:3081/    # API — returns JSON
curl -I  http://localhost:3082/    # Dashboard — returns 200
```

- Dashboard: http://localhost:3082
- API: http://localhost:3081
- MinIO Console: http://localhost:9001 (default login: `minioadmin` / `minioadmin`)

### Run from Pre-built Images (no build)

Building from source needs the full repo and ~8 GB RAM for the dashboard build. To skip that, pull
the published `classroomio/{api,dashboard,jobs}` images instead — you only need the images compose
file and a `.env`:

```zsh title="Terminal"
curl -O https://raw.githubusercontent.com/classroomio/classroomio/main/docker/docker-compose.images.yaml
curl -o .env https://raw.githubusercontent.com/classroomio/classroomio/main/.env.example
# Edit .env: set CIO_VERSION (pin a release), DASHBOARD_ORIGIN, secrets, SMTP
CIO_VERSION=1.4.2 docker compose -f docker-compose.images.yaml --env-file .env pull
CIO_VERSION=1.4.2 docker compose -f docker-compose.images.yaml --env-file .env up -d
```

<Callout title="Pin a version in production">
  `CIO_VERSION` chooses the image tag. Pin an exact release (e.g. `1.4.2`) so a new release never
  breaks you unattended; `latest` = newest release, `edge` = bleeding edge. Rollback = set
  `CIO_VERSION` back and re-pull. From a full clone, `./run-docker-full-stack.sh --images` does the
  same. Note: secrets are NOT auto-generated on this path — set `BETTER_AUTH_SECRET`,
  `AUTH_BEARER_TOKEN`, and `PRIVATE_SERVER_KEY` (last two identical) yourself.
</Callout>

### One `.env`, One Command

Docker self-hosting uses a single root `.env` file. Every `docker compose` command reads it via `--env-file .env` — you never need separate env files per service. See [`.env.example`](https://github.com/classroomio/classroomio/blob/main/.env.example) for the full template.

### Environment Variables

#### Required

| Variable | Description | Local Default |
|----------|-------------|---------------|
| `POSTGRES_DB` | Database name | `classroomio` |
| `POSTGRES_USER` | Database user | `postgres` |
| `POSTGRES_PASSWORD` | Database password | `postgres` |
| `BETTER_AUTH_SECRET` | Auth session signing secret | — |
| `AUTH_BEARER_TOKEN` | API bearer token | Auto-generated |
| `PRIVATE_SERVER_KEY` | Dashboard-to-API auth key (same in both) | Auto-generated |
| `PUBLIC_IS_SELFHOSTED` | Self-hosted mode flag | `true` |
| `PRIVATE_SERVER_URL` | Internal API URL for SSR (Docker network) | `http://api:3081` |
| `DASHBOARD_ORIGIN` | Dashboard origin for invite links, email links, etc. | `http://localhost:3082` |

#### Optional — Direct API Access

Dashboard browser requests use same-origin `/proxy/*` and `/api/auth/*` paths, then the dashboard forwards them to `PRIVATE_SERVER_URL`. Set these only when browsers or third-party clients need to call the API origin directly.

| Variable | Description | Local Default |
|----------|-------------|---------------|
| `PUBLIC_SERVER_URL` | Canonical public API URL / direct API fallback | `http://localhost:3081` |
| `TRUSTED_ORIGINS` | Comma-separated CORS origins for direct browser-to-API calls | — |

#### CSP Domains (Runtime)

CSP domain variables are **runtime** — set them in your `.env` or `docker-compose.yaml` `environment:` section. No image rebuild is needed when changing them.

| Variable | Description |
|----------|-------------|
| `ALLOWED_EXTERNAL_DOMAINS` | Overrides all per-directive CSP vars (comma-separated) |
| `CSP_SCRIPT_SRC_DOMAINS` | Domains allowed in `script-src` |
| `CSP_STYLE_SRC_DOMAINS` | Domains allowed in `style-src` |
| `CSP_CONNECT_SRC_DOMAINS` | Domains allowed in `connect-src` |
| `CSP_FRAME_SRC_DOMAINS` | Domains allowed in `frame-src` |
| `CSP_FONT_SRC_DOMAINS` | Domains allowed in `font-src` |
| `CSP_MEDIA_SRC_DOMAINS` | Domains allowed in `img-src` and `media-src` |

<Callout type="warn" title="CSP Domains">
  When `PUBLIC_IS_SELFHOSTED=true`, the dashboard starts with only `'self'` allowed. Set `ALLOWED_EXTERNAL_DOMAINS` (or the per-directive vars) with your CDN, video, and font domains. The API does not need to be added for normal dashboard traffic because requests are proxied same-origin. Since these are runtime env vars, you can change them by restarting the container — no rebuild required.
</Callout>

#### Optional — Email

Configure either SMTP or Zoho. Without email config, the app works but cannot send emails.

| Variable | Description |
|----------|-------------|
| `SMTP_HOST`, `SMTP_PORT`, `SMTP_USER`, `SMTP_PASSWORD`, `SMTP_SENDER` | SMTP server |
| `ZOHO_TOKEN` | Zoho/ZeptoMail token (alternative to SMTP) |

#### Optional — Object Storage

MinIO is included by default. The startup script auto-configures these — you only need to change them if using custom credentials or an external S3-compatible service.

| Variable | Default |
|----------|---------|
| `MINIO_ROOT_USER` | `minioadmin` |
| `MINIO_ROOT_PASSWORD` | `minioadmin` |
| `OBJECT_STORAGE_ENDPOINT` | `http://minio:9000` |
| `OBJECT_STORAGE_PUBLIC_ENDPOINT` | `http://localhost:9000` |
| `OBJECT_STORAGE_ACCESS_KEY_ID` | `minioadmin` |
| `OBJECT_STORAGE_SECRET_ACCESS_KEY` | `minioadmin` |
| `OBJECT_STORAGE_FORCE_PATH_STYLE` | `true` |
| `OBJECT_STORAGE_MEDIA_PUBLIC_BASE_URL` | `http://localhost:9000/media` |

Use `./run-docker-full-stack.sh --no-minio` to exclude MinIO if you do not need uploads.

<Callout type="error" title="Change default credentials">
  Do not use `minioadmin` / `minioadmin` in production.
</Callout>

#### Optional — Integrations

| Variable | Description |
|----------|-------------|
| `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET` | Google OAuth |
| `PRIVATE_APP_HOST`, `PRIVATE_APP_SUBDOMAINS` | Multi-subdomain routing |
| `UNSPLASH_API_KEY` | Unsplash image search |
| `LICENSE_KEY` | Enterprise features (SSO, token-auth, no-tracking) |
| `PUBLIC_APP_TITLE`, `PUBLIC_APP_DESCRIPTION`, `PUBLIC_OG_IMAGE_URL` | Custom branding |

### Common Commands

```zsh title="Terminal"
# Start full stack (builds images, starts all services including MinIO)
./run-docker-full-stack.sh

# Start without rebuilding images
./run-docker-full-stack.sh --no-build

# Start without MinIO (no file uploads)
./run-docker-full-stack.sh --no-minio

# Service status
docker compose --env-file .env -p classroomio -f docker-compose.yaml ps

# Stream logs
docker compose --env-file .env -p classroomio -f docker-compose.yaml logs -f api dashboard

# Rebuild one service
docker compose --env-file .env -p classroomio -f docker-compose.yaml up -d --build api

# Restart after env change (no rebuild needed)
docker compose --env-file .env -p classroomio -f docker-compose.yaml restart api dashboard

# Run DB seed (schema setup already runs on api startup; this adds demo data)
docker exec cio-api pnpm --filter @cio/db db:setup:seed

# Stop
docker compose --env-file .env -p classroomio -f docker-compose.yaml down

# Stop and delete all data (database, redis, minio volumes)
docker compose --env-file .env -p classroomio -f docker-compose.yaml down -v
```

See [docker/docs/SELF_HOST.md](https://github.com/classroomio/classroomio/blob/main/docker/docs/SELF_HOST.md) for the full Docker reference including troubleshooting.

### Troubleshooting: "No space left on device"

```zsh title="Terminal"
docker system df              # check usage
docker builder prune -f       # free build cache
docker image prune -a -f      # free unused images
docker container prune -f     # free stopped containers
```

<Callout type="warn" title="Volume prune deletes data">
  Avoid `docker volume prune` after `docker compose ... down` — it can remove your database and storage volumes.
</Callout>

Then restart: `./run-docker-full-stack.sh`

## Railway

The quickest hosted deployment is the [Railway Template](https://railway.com/deploy/classroomio-selfhost?referralCode=8OyIj5).

[![Deploy on Railway](https://railway.com/button.svg)](https://railway.com/deploy/classroomio-selfhost?referralCode=8OyIj5)

A complete deployment needs **three** app services — `cio-api`, `cio-dashboard`, and the
`cio-jobs` worker — plus Railway's **Postgres** and **Redis** plugins. Each app service builds from
this repo; set its config path to the matching `railway.*.json` (`railway.api.json`,
`railway.dashboard.json`, `railway.jobs.json`) so Railway uses the correct Dockerfile.

<Callout type="warn" title="Don't skip the jobs worker">
  `cio-jobs` is **required**. Without it, video processing, transcription, AI course generation, and
  most emails are queued but never run — uploads stay stuck with no error. If your template only has
  api + dashboard, add the worker service below.
</Callout>

### cio-api

```zsh title="env"
DATABASE_URL="${{db.DATABASE_URL}}"
REDIS_URL="${{Redis.REDIS_URL}}"
BETTER_AUTH_SECRET="random-secure-key"
AUTH_BEARER_TOKEN="some-key"
PRIVATE_SERVER_KEY="same-key-for-api-and-dashboard"
PUBLIC_IS_SELFHOSTED="true"

# Dashboard URL for invite/email links (e.g. https://app.yourdomain.com)
DASHBOARD_ORIGIN="https://app.yourdomain.com"

# Optional direct public API URL. Normal dashboard auth/API calls use the
# dashboard proxy and do not require API/dashboard cookie-domain matching.
PUBLIC_SERVER_URL="https://api.yourdomain.com"
TRUSTED_ORIGINS=

# Email (configure one)
SMTP_HOST=
SMTP_PORT=
SMTP_USER=
SMTP_PASSWORD=
SMTP_SENDER=

# Object storage — Cloudflare R2 (for hosted deployments)
CLOUDFLARE_ACCOUNT_ID=
CLOUDFLARE_ACCESS_KEY=
CLOUDFLARE_SECRET_ACCESS_KEY=
CLOUDFLARE_BUCKET_DOMAIN=
CLOUDFLARE_IMAGE_BUCKET_DOMAIN=
```

Optional: add `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET` for OAuth, `UNSPLASH_API_KEY` for image search, or `LICENSE_KEY` for enterprise features.

### cio-dashboard

```zsh title="env"
ALLOWED_EXTERNAL_DOMAINS="https://*.yourdomain.com,https://fonts.googleapis.com"
PRIVATE_APP_HOST="https://app.yourdomain.com"
PRIVATE_SERVER_KEY="same-key-for-api-and-dashboard"
PUBLIC_IS_SELFHOSTED="true"
PRIVATE_SERVER_URL="http://${{api.RAILWAY_PRIVATE_DOMAIN}}:8080"
ORIGIN="https://app.yourdomain.com"
```

Optional: add `PUBLIC_APP_TITLE`, `PUBLIC_APP_DESCRIPTION`, `PUBLIC_OG_IMAGE_URL` for custom branding. You can also use per-directive CSP vars (`CSP_SCRIPT_SRC_DOMAINS`, `CSP_MEDIA_SRC_DOMAINS`, etc.) instead of `ALLOWED_EXTERNAL_DOMAINS` for finer control.

### cio-jobs (background worker)

Build this service from `docker/Dockerfile.jobs` (use `railway.jobs.json`). It has **no public domain**
— it's a worker that drains the Redis queue. It shares the same Postgres, Redis, storage, email, and
OpenAI settings as the API:

```zsh title="env"
DATABASE_URL="${{db.DATABASE_URL}}"
REDIS_URL="${{Redis.REDIS_URL}}"

# Object storage — same Cloudflare R2 bucket as the API
CLOUDFLARE_ACCOUNT_ID=
CLOUDFLARE_ACCESS_KEY=
CLOUDFLARE_SECRET_ACCESS_KEY=
CLOUDFLARE_IMAGE_BUCKET_DOMAIN=

# Email (worker sends invites, notifications, etc.)
SMTP_HOST=  SMTP_PORT=  SMTP_USER=  SMTP_PASSWORD=  SMTP_SENDER=

# Transcription + OpenAI-backed AI course generation
OPENAI_API_KEY=
```

<Callout title="PRIVATE_SERVER_KEY">
  `PRIVATE_SERVER_KEY` must be the same value in both API and dashboard. It authenticates server-side dashboard requests to the API.
</Callout>

### SMTP Setup

You need SMTP for email sending. See [Nodemailer SMTP docs](https://nodemailer.com/smtp/) for provider setup.

### Cloudflare R2

For hosted deployments, Cloudflare R2 handles object storage. See the [Cloudflare Setup](/docs/contributor-guides/cloudflare-setup) guide.

### SSO (Enterprise)

SSO requires a license key. Set `LICENSE_KEY` in your API environment. [Book a call](http://cal.com/classroomio/enterprise) to get a license.

## Coolify

[Coolify](https://coolify.io) is a self-hosted PaaS (an open-source Heroku/Netlify alternative) you
run on your own VPS. It deploys ClassroomIO as a **Docker Compose resource** using the pull-based
[`docker-compose.images.yaml`](https://github.com/classroomio/classroomio/blob/main/docker/docker-compose.images.yaml) —
so your Coolify host **pulls** the published `classroomio/*` images and never builds anything.

### Prerequisites

- A running Coolify instance with a connected server.
- A domain pointed at that server (Coolify's built-in proxy terminates HTTPS for you).

### Steps

1. **New Resource → Docker Compose.** Point Coolify at this repo and the compose path
   `docker/docker-compose.images.yaml`, or paste the file's contents.

2. **Set environment variables** (Coolify's UI — these become the compose `.env`). Coolify does
   **not** run our `run-docker-full-stack.sh`, so generate secrets yourself:

   ```zsh title="Required"
   CIO_VERSION=1.4.2                 # pin a release (not "latest") so upgrades are deliberate
   DASHBOARD_ORIGIN=https://app.yourdomain.com
   BETTER_AUTH_SECRET=<openssl rand -hex 32>
   AUTH_BEARER_TOKEN=<openssl rand -hex 32>
   PRIVATE_SERVER_KEY=<same value as AUTH_BEARER_TOKEN>
   POSTGRES_PASSWORD=<strong password>
   # Email (required for signup/reset/invites)
   SMTP_HOST=  SMTP_PORT=  SMTP_USER=  SMTP_PASSWORD=  SMTP_SENDER=
   ```

3. **Assign a domain.** Map your domain to the **dashboard** service (internal port `3082`); Coolify's
   reverse proxy issues the certificate. The dashboard proxies API calls server-side, so you normally
   only need the dashboard exposed publicly — the `api` service stays internal.

4. **Object storage.** Recommended: external **Cloudflare R2 / S3** — set the `OBJECT_STORAGE_*`
   variables and you're done (no stateful storage container to manage on Coolify). To use the bundled
   MinIO instead, enable its profile (`COMPOSE_PROFILES=minio`); note it needs a persistent volume and
   you must route its public URL.

5. **Deploy.** Coolify pulls `classroomio/{api,dashboard,jobs}:${CIO_VERSION}`, starts Postgres + Redis,
   the API container runs database setup on first boot, then the dashboard and the **jobs worker** come up.

<Callout title="The jobs worker is included">
  The compose ships the `jobs` worker — required for video processing, transcription, AI course
  generation, and most emails. Without it those tasks queue forever. (The older Railway template
  predates this; make sure your deployment runs all three services.)
</Callout>

### Upgrades & rollback

Bump `CIO_VERSION` to a newer release and redeploy; roll back by setting it to the previous version
and redeploying. Because you pin a version, a new release never restarts you onto broken images
unattended.
